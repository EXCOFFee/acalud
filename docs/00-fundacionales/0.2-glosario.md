# 0.2 · Glosario — Lenguaje ubicuo de Acalud

| Campo | Valor |
|---|---|
| **Artefacto** | 0.2 Glosario / Lenguaje ubicuo |
| **Versión** | 0.1.0 |
| **Fecha** | 2026-07-04 |
| **Estado** | 🟡 Borrador — pendiente ratificación del equipo |

**Fundamento:** siguiendo la práctica de *Lenguaje Ubicuo* del Domain-Driven Design (Evans,
`Evans03.pdf` de la base de conocimiento del proyecto), cada término tiene **un único
significado** dentro del dominio, y ese significado se usa idéntico en documentación, código
(nombres de clases, tablas, endpoints) y conversación. Si el equipo detecta que un término se
usa con dos sentidos, eso es un hallazgo de diseño: se registra y se resuelve, no se convive.

**Regla de estilo:** en documentación y código queda **prohibido el término genérico
"usuario"** salvo en "Usuario sin login (USL)". Siempre se nombra el actor específico
(Docente, Encargado institucional, Admin). Evita ambigüedades de permisos y de modelo.

## A. Términos del dominio

| Término | Definición | Notas / decisiones vinculadas |
|---|---|---|
| **Admin Acalud** | Operador del backoffice de la editorial. Gestiona catálogo, stock, pedidos, encuestas, propuestas y descuentos. | Único actor con MFA obligatorio (S-06). |
| **Carrito** | Colección temporal de ítems (juego físico + cantidad) de un Docente antes del checkout. | El descuento mayorista se aplica por reglas de cantidad (CU-010, CU-022). |
| **Checkout** | Proceso que convierte un Carrito en un Pedido: dirección, cotización de envío, pago. | Flujo crítico; vitrina L3 (artefacto 3.2). |
| **Comprobante** | Documento que certifica una compra. Dos variantes: **PDF interno** (siempre) y **comprobante ARCA de homologación** (cuando el adaptador está activo). | S-07: se emite tras pago aprobado. Sin validez fiscal (R-03). |
| **CAE** | Código de Autorización Electrónico que ARCA otorga a un comprobante. En homologación, carece de validez legal. | Ver 9.3 de Visión y Alcance. |
| **Demo** | Versión de prueba de un juego, accesible online. **Demo pública**: sin login (CU-006). **Demo registrada**: requiere cuenta de Docente (CU-007). | La diferencia exacta de contenido entre ambas se define en 1.1. |
| **Descuento mayorista** | Regla de precio por cantidad configurada por el Admin (CU-022) y aplicada automáticamente en el Carrito (CU-010). | Aplica tanto a Docentes como a compras institucionales de Lotes. |
| **Desvinculación** | Baja de la Membresía de un Docente en una Institución (CU-028). No borra su historial de Sesiones de uso (S-03). | El Docente conserva su cuenta y sus compras personales. |
| **Docente** | Persona registrada con cuenta individual. Compra, descarga recursos, participa en comunidad. | Identidad base del sistema (S-01). |
| **Docente institucional** | **Rol** que adquiere un Docente cuando tiene una Membresía activa en una Institución. No es una cuenta separada. | S-01/S-02. Habilita CU-029/030. |
| **Editorial aliada** | Editorial tercera exhibida en la vitrina (CU-017). **No vende** a través de Acalud. | Decisión de alcance: vitrina, sin transacciones. |
| **Encargado institucional** | Representante de una Institución. Registra la institución, compra Lotes, administra Membresías, consulta reportes y dashboard. | CU-023..028, 031..033. |
| **Encuesta** | Instrumento de feedback parametrizado por el Admin (CU-020) y respondido por Docentes (CU-014), con resultados visibles (CU-016). | La visibilidad de resultados (quién ve qué) se detalla en 1.1. |
| **Envío** | Traslado físico de un Pedido. Tiene costo cotizado (CU-011) y seguimiento (CU-013). | Cotización vía MiCorreo con fallback a Tabla de tarifas. |
| **Favorito** | Marca que un Docente pone sobre una Editorial aliada o su contenido (CU-018). | Alcance exacto (¿editorial, producto exhibido o ambos?) se cierra en 1.1. |
| **Institución** | Organización educativa registrada (CU-023) que compra Lotes y agrupa Docentes mediante Membresías. | |
| **Juego** | Producto físico del catálogo de Acalud. Puede tener Demos y Recursos asociados. | Único tipo de producto vendible en v1 (S-04). |
| **Lote** | Compra institucional de juegos físicos en cantidad (CU-024). | Usa las mismas reglas de Descuento mayorista y Envío. |
| **Membresía** | Relación Docente↔Institución con estado (activa/desvinculada) e historial. | Muchos-a-muchos (S-02). Creada por invitación o registro directo (CU-026). |
| **Pedido** | Resultado de un Checkout pagado: ítems, dirección, envío, estado del ciclo de vida (pagado → en preparación → despachado → entregado). | Gestión del ciclo: CU-E03 (Admin). |
| **Propuesta de juego** | Idea de juego enviada por un Docente (CU-015) y revisada por el Admin (CU-021) con un estado (recibida/en revisión/aceptada/rechazada). | |
| **Recurso libre** | Material digital descargable sin restricción (CU-008), con o sin login. | |
| **Recurso licenciado** | Material digital que **se desbloquea al comprar el Juego asociado** (CU-009). No se vende por separado. | S-04. Implica control de autorización por recurso. |
| **Sesión de uso** | Registro que un Docente institucional carga sobre el uso de un Juego físico en el aula (CU-029): fecha, juego, y métricas pedagógicas a definir en 1.1. | Alimenta reportes (CU-031) y dashboard (CU-033). |
| **Stock** | Cantidad disponible de un Juego. Se descuenta al confirmar la compra; sin stock no se compra. | S-05. Gestión: CU-E04. |
| **Tracking** | Número y eventos de seguimiento de un Envío. | Provisto por MiCorreo o cargado a mano por el Admin (fallback). |
| **USL** | Usuario sin login. Navega, prueba demos públicas, descarga recursos libres, explora la vitrina. | Puede registrarse (CU-001). |

## B. Términos técnicos del proyecto

| Término | Definición en este proyecto |
|---|---|
| **ADR** | *Architecture Decision Record*: documento corto que registra una decisión de arquitectura, sus alternativas, y sus consecuencias. Viven en el Bloque 2.2. |
| **Adaptador** | Implementación concreta de un Puerto para un proveedor específico (ej.: `MiCorreoAdapter`, `TarifaLocalAdapter`). |
| **Bounded Context** | Frontera de significado del modelo de dominio (DDD). Se formalizan en 2.3. |
| **Compuerta de calidad** | Conjunto de criterios verificables (tests, checklists) que una etapa del plan 5.2 debe pasar antes de habilitar la siguiente. |
| **Fallback** | Comportamiento degradado y controlado cuando una integración externa falla (ej.: Tabla de tarifas si MiCorreo no responde). |
| **Homologación (ARCA)** | Ambiente de prueba de los web services de ARCA; comportamiento idéntico a producción, comprobantes sin validez legal. |
| **MECE** | Mutuamente Excluyente, Colectivamente Exhaustivo: propiedad del índice de artefactos. |
| **Puerto** | Interfaz propia del dominio que abstrae un servicio externo (`PaymentProvider`, `ShippingProvider`, `ReceiptProvider`). El dominio depende del puerto, nunca del proveedor. |
| **Sandbox (Mercado Pago)** | Ambiente de prueba de MP: tarjetas y usuarios ficticios, sin dinero real. |
| **Tabla de tarifas** | Estructura local (peso × zona → precio) usada como fallback de cotización de envíos. |
| **Vitrina L3** | Sección de la documentación donde un flujo (checkout) se analiza con profundidad máxima como demostración de método. |
| **WSAA / WSFEv1** | Web services de ARCA: autenticación (ticket de acceso) y facturación electrónica, respectivamente. |
