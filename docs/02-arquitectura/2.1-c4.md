```mermaid
graph TB

    subgraph ACALUD ["ACALUD — plataforma de juegos educativos"]
        SYS["Sistema Acalud<br/>Catálogo · Checkout · Instituciones ·<br/>Métricas pedagógicas · Comunidad · Backoffice"]
    end

    MP["🌐 Mercado Pago<br/>(sandbox)"]
    MC["🌐 Correo Argentino<br/>MiCorreo (test)"]
    AR["🌐 ARCA WSFEv1<br/>(homologación)"]
    EM["🌐 Proveedor de email<br/>(free tier)"]
    MON["🌐 Monitor sintético<br/>(UptimeRobot)"]

    USL --> SYS
    DOC --> SYS
    ENC --> SYS
    ADM --> SYS
    SYS -->|cobra pedidos| MP
    MP -->|webhook de pago| SYS
    SYS -->|cotiza y trackea| MC
    SYS -->|solicita CAE de prueba| AR
    SYS -->|envía transaccionales| EM
    MON -->|/health cada 5 min<br/>SLO + keep-alive| SYS
```

**Lectura:** cuatro actores humanos, cinco sistemas externos. La única entrada asíncrona
externa es el webhook de MP (vector de seguridad prioritario para 3.1). El monitor cumple
doble función (NFR-D1 + anti-spin-down, ADR-005).

```mermaid
graph TB

        API["⚙️ API Monolito Modular<br/>NestJS · hexagonal (ADR-002)<br/>7 módulos por bounded context"]
        JOBS["⏱️ Scheduler + Outbox worker<br/>in-process<br/>(expiración PC-03 · emails PG-03 · consistencia stock)"]
    end

    subgraph DATOS ["Supabase Free (ADR-003)"]
        PG[("🗄️ PostgreSQL<br/>vía pooler")]
        ST[("📦 Storage<br/>recursos · comprobantes ·<br/>imágenes · adjuntos")]
    end

    WEB -->|"/api/* (rewrite Vercel)<br/>cookie httpOnly same-site"| API
    APK -->|"HTTPS directo<br/>Authorization: Bearer"| API
    API --> PG
    API -->|enlaces firmados PD-04| ST
    JOBS --> PG
    API -->|puertos ADR-006| EXT["MP · MiCorreo · ARCA · Email"]
```

**Decisiones visibles acá:** transporte dual de sesión (ADR-004); worker y jobs dentro del
proceso (ADR-005: no hay horas para un segundo servicio); el storage nunca se expone
directo — siempre enlaces firmados de corta vida (PD-04).

## Nivel 3 — Componentes del módulo Compras (el crítico)

```mermaid
graph TB
    subgraph PRIMARIOS ["Adapters primarios (driving)"]
        CTRL["CheckoutController<br/>REST"]
        WH["WebhookMPController<br/>+ verificación de firma"]
        SCHED["ExpiracionJob<br/>(PC-03)"]
    end

    subgraph APP ["Application — casos de uso"]
        IDEM["IdempotencyHandler<br/>payment_id UNIQUE"]
        CMD1["IniciarCheckoutCommand<br/>(Unit of Work)"]
        CMD2["ProcesarPagoCommand<br/>(Unit of Work)"]
        STRAT["Strategy: contexto<br/>personal | institucional"]
    end

    subgraph DOM ["Domain"]
        PED["Pedido (Aggregate Root)<br/>máquina de estados + snapshot"]
        STK["Stock / Kardex<br/>decremento condicional"]
        PP["Puertos: PaymentProvider ·<br/>ShippingProvider · ReceiptProvider"]
    end

    subgraph SEC ["Adapters secundarios (driven)"]
        REPO["PedidoRepository (PG)"]
        MPA["MercadoPagoSandboxAdapter"]
        SHA["MiCorreoAdapter ⇄ TarifaLocalAdapter<br/>(circuit breaker)"]
        REC["PdfInterno + ArcaHomologacion<br/>(vía outbox PC-06)"]
        OUT["Outbox (emails + eventos)"]
    end

    CTRL --> CMD1 --> STRAT --> PED
    WH --> IDEM --> CMD2 --> PED
    CMD2 --> STK
    SCHED --> PED
    PED --> PP
    PP --> MPA & SHA & REC
    CMD1 & CMD2 --> REPO
    CMD2 --> OUT
```

**Regla de dependencias verificada en el dibujo:** las flechas de dominio salen solo hacia
puertos; ningún componente de `DOM` conoce un adapter concreto. `ProcesarPagoCommand` es la
transacción única del paso 5 de CU-012: transición de estado + decremento multi-línea +
outbox, todo-o-nada.

**Los otros seis módulos** (Identidad, Catálogo, Logística, Comprobantes, Comunidad,
Institucional) replican esta anatomía; sus componentes se enumeran en 2.3 junto a sus
Aggregates — repetir seis diagramas idénticos sería ruido, no información.
