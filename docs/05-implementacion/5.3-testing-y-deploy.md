## 1. Estrategia de testing (operacionaliza la pirámide de ADR-002)

| Nivel | % objetivo | Herramienta | Qué cubre | Regla dura |
|---|---|---|---|---|
| Unit | ~70 % | Vitest/Jest | Dominio puro: invariantes de Aggregates (2.3 §5), cálculo de precios/descuentos, máquinas de estado | Prohibido mockear clases de dominio propias |
| Integración | ~20 % | Testcontainers + PostgreSQL real | Repos, constraints, **idempotencia** (payment_id), **IDOR** (recurso ajeno→404), transacciones multi-línea | Prohibida BD in-memory |
| Contract | ~5 % | Schemathesis/Dredd sobre openapi.yaml | Que la API cumpla la spec; errores Problem Details válidos | La spec es la fuente |
| E2E | ~5 % | Playwright | Los flujos de CE-01, en web y (smoke) en APK | ≤ 15 flujos, los críticos |
| Performance | — | k6 | NFR-C2 a 3× ráfaga con thresholds como criterio de fallo | Threshold rojo = build roja |

**Escenarios Gherkin de 1.1 = criterio de aceptación:** cada CU trae sus escenarios; el CU
no está hecho hasta que están en verde. Los escenarios `@seguridad` (IDOR, anti-enumeración,
idempotencia) son **obligatorios en CI** (NFR-S7).

**Datos de prueba:** factories que generan datos ficticios válidos (CUIT con dígito
verificador, emails, domicilios AR) — R-04 "se diseña como real". El seed de demo (NFR-C3) es
un superconjunto reproducible.

## 2. Deploy web (free tier, $0)

**Frontend (Vercel Hobby):** push a `main` → build automático de Next.js (static export) →
publicación. Los rewrites `/api/*` (ADR-004) se configuran en `vercel.json`. Preview
deployments por PR (gratis) sirven para revisar antes de mergear.

**Backend (Render free):** deploy desde GitHub → build de NestJS → servicio web con las
variables de entorno (secretos: credenciales MP, token MiCorreo, certificado ARCA, DB URL,
email API key). Health check en `/health`. **Un solo servicio** (regla dura ADR-005): el
worker de outbox y el scheduler corren in-process.

**BD/Storage (Supabase free):** migraciones aplicadas por script versionado (`infra/`) antes
de habilitar tráfico; nunca a mano.

**Keep-alive:** UptimeRobot (free) sobre `/health` y `/ready` cada 5 min — mide SLO y evita
spin-down de Render + pausa de Supabase (ADR-005).

**Pipeline de promoción:**
```
PR → CI (8 gates 5.1) + preview Vercel → revisión humana → merge a main
   → deploy auto web (Vercel) + api (Render) → smoke test post-deploy (/health, 1 flujo E2E)
```

## 3. Generación de la APK (sideload, sin store)

**Proceso 100 % local (ADR-001), $0, sin cuentas de build en la nube:**
1. `next build` (static export) genera el bundle web.
2. `npx cap sync android` copia el bundle al proyecto Android nativo.
3. Build de release con Gradle (`./gradlew assembleRelease`) desde Android Studio o CLI.
4. **Firma** con un keystore propio generado una vez (`keytool`); la clave del keystore es un
   secreto del equipo (no va al repo).
5. La APK firmada se publica como **asset en GitHub Releases** con su **hash SHA-256**
   publicado (mitigación de suplantación — 3.1 V8).

**Distribución:** el usuario descarga la APK del Release y la instala habilitando "orígenes
desconocidos" (inherente al sideload R-07; documentado). Sin Play Store, sin revisión de
tienda, sin costos.

**Verificación de paridad:** el mismo bundle corre en web y APK; los smoke tests E2E
(Playwright web + verificación manual guiada en APK por checklist NFR-X1/X2) confirman que
los flujos críticos funcionan idénticos en ambos.

## 4. Versionado y releases

- **Git:** trunk-based con PRs cortos (una tarea de 5.2 = un PR revisable). `main` siempre
  desplegable.
- **Migraciones:** hacia adelante, versionadas, idempotentes donde se pueda; nunca editar una
  migración ya aplicada (se agrega una nueva).
- **Releases:** un tag por hito de 5.2; el asset APK acompaña al tag; el CHANGELOG resume qué
  CU entraron.
- **Rollback:** web y api redeployan la versión anterior desde el proveedor; la BD se protege
  con el backup diario + export semanal (NFR-R1) y el restore ensayado (NFR-R2).
